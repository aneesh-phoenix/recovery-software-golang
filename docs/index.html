A Go-based file recovery tool that recovers PDF and ZIP files from formatted or corrupted disks, with full NTFS and ext4 filesystem awareness.

What is File Recovery?

How Deletion Actually Works

When you delete a file on your computer, the data is not immediately erased from the disk. Instead, the operating system simply marks the space as "available for reuse." Think of it like a library removing a book's entry from the catalog — the book is still on the shelf, but nobody knows where to find it.

At the disk level, here's what happens:

  1. Filesystem metadata is updated — The directory entry or inode is marked as deleted
  2. Data blocks are freed — The blocks are added to the "free space" pool
  3. Actual data remains — The raw bytes on disk are untouched until overwritten
⚠️ Important

The longer you wait after deletion, the higher the chance that new data will overwrite the old file blocks. Run recovery tools as soon as possible after data loss.

What is File Carving?

File carving is a recovery technique that scans raw disk bytes looking for known file patterns — without relying on any filesystem metadata. It's like searching through a pile of shredded documents by recognizing the header and footer of each page.

Every file format has unique identifiers called magic bytes (or file signatures):

  • %PDF (hex: 25 50 44 46) marks the start of a PDF file
  • PK\x03\x04 (hex: 50 4B 03 04) marks the start of a ZIP file
  • %%EOF marks the end of a PDF file

The carver reads disk data block-by-block, checking each block for these signatures. When a header is found, it continues reading until the corresponding footer is located, then extracts everything in between as a recovered file.

Filesystem Metadata Recovery

While carving works on any disk regardless of filesystem state, filesystem-aware recovery can produce much better results when the metadata structures are still partially intact.

NTFS (Windows) uses a Master File Table (MFT) — a database-like structure that records every file's name, size, timestamps, and disk locations. Even after formatting, MFT entries often survive.

ext4 (Linux) uses inodes — data structures that store file metadata and pointers to data blocks. Deleted inodes retain much of this information, including deletion timestamps.

💡 Key Insight

Filesystem-aware recovery can restore original filenames and directory structures, while raw carving can only produce generically-named files (e.g., recovered_001.pdf).

How the Pieces Fit Together

The architecture follows a layered design where each package has a single responsibility:

  1. cmd/recover/main.go — The CLI parses command-line arguments (mode, device path, output directory) and dispatches to the appropriate recovery engine.
  2. pkg/scanner — Provides the low-level foundation: opening devices, reading blocks at arbitrary offsets, and iterating over entire disks in fixed-size chunks.
  3. pkg/signature — Defines what files look like at the byte level. The carver and filesystem parsers both use these signatures to identify file types.
  4. pkg/carver — The raw carving engine. Uses the scanner to read blocks and signatures to detect files. This is the filesystem-independent fallback.
  5. pkg/ntfs & pkg/ext4 — Filesystem-specific parsers that understand metadata structures. They extract filenames, sizes, and data locations from MFT entries (NTFS) or inodes (ext4).
  6. pkg/output — Handles writing recovered files to disk and generating summary reports.
💡 Design Philosophy

Each package can be tested independently. The scanner doesn't know about file types. The signatures don't know about disk I/O. This separation makes the code easier to understand, test, and extend.

How the Scanner Works

Overview

The scanner package (pkg/scanner/scanner.go) provides the foundation for all disk access in recovery-soft. It abstracts away the complexities of reading from block devices and provides a clean, thread-safe interface for the higher-level recovery engines.

The DiskReader Struct

At the core of the scanner is the DiskReader struct, which wraps Go's standard os.File with additional metadata and synchronization:

type DiskReader struct {
    file      *os.File       // The underlying file or block device
    size      int64          // Total size in bytes
    blockSize int            // Read chunk size (default: 4096 bytes)
    mu        sync.Mutex     // Protects concurrent access to file handle
}

The struct stores the file handle, pre-calculated size, configurable block size, and a mutex for thread safety. The block size of 4096 bytes matches the most common filesystem allocation unit size, which means each read aligns with potential file boundaries.

NewDiskReader — Opening a Device

The constructor handles both regular files (like disk images) and actual block devices:

func NewDiskReader(path string, blockSize int) (*DiskReader, error) {
    file, err := os.Open(path)
    if err != nil {
        return nil, fmt.Errorf("failed to open device: %w", err)
    }

    // Get file size - for block devices, Seek to end
    size, err := file.Seek(0, io.SeekEnd)
    if err != nil {
        return nil, fmt.Errorf("failed to determine size: %w", err)
    }

    // Reset position to start
    _, err = file.Seek(0, io.SeekStart)

    return &DiskReader{
        file:      file,
        size:      size,
        blockSize: blockSize,
    }, nil
}

Why use Seek for size? Block devices (like /dev/sda) don't support the typical Stat() call for getting size. Instead, we seek to the end of the device — the offset returned by Seek(0, io.SeekEnd) gives us the total byte count. This works universally for both regular files and block devices.

ReadAt — Thread-Safe Random Access

The ReadAt method allows any goroutine to read from any offset safely:

func (dr *DiskReader) ReadAt(buf []byte, offset int64) (int, error) {
    dr.mu.Lock()
    defer dr.mu.Unlock()

    _, err := dr.file.Seek(offset, io.SeekStart)
    if err != nil {
        return 0, err
    }
    return dr.file.Read(buf)
}

The mutex ensures that the Seek + Read pair executes atomically. Without this lock, two goroutines could interleave their seeks and reads, causing each to read from the wrong offset. This pattern is essential because Go's os.File maintains a single internal cursor position.

💡 Why not use os.File.ReadAt?

Go's standard library does provide a ReadAt method on os.File that's inherently thread-safe. This custom implementation gives us more control over error handling and allows adding instrumentation (like read counters or progress callbacks) in a single place.

ScanBlocks — Iterating Over the Disk

The workhorse method that drives the carving process:

func (dr *DiskReader) ScanBlocks(callback func(offset int64, data []byte) error) error {
    buf := make([]byte, dr.blockSize)

    for offset := int64(0); offset < dr.size; offset += int64(dr.blockSize) {
        n, err := dr.ReadAt(buf, offset)
        if err != nil && err != io.EOF {
            return err
        }
        if n == 0 {
            break
        }

        if err := callback(offset, buf[:n]); err != nil {
            return err
        }
    }
    return nil
}

How it works step by step:

  1. Allocates a reusable buffer of blockSize bytes (avoids repeated allocation)
  2. Iterates from offset 0 to the end of the disk, advancing by blockSize each iteration
  3. Reads each block into the buffer using ReadAt
  4. Passes the offset and data to the callback function (provided by the carver)
  5. The callback can return an error to stop scanning early

The callback pattern lets the scanner remain ignorant of what the caller is looking for. The carver passes a callback that checks each block for magic bytes. The NTFS parser passes a different callback that looks for MFT signatures.

File Signatures Explained

What Are Magic Bytes?

Magic bytes (also called file signatures) are fixed sequences of bytes at the beginning of a file that identify its format. They're called "magic" because early Unix systems used a file called /etc/magic to store these patterns for the file command.

Here are the signatures recovery-soft looks for:

File Type Header (ASCII) Header (Hex) Footer
PDF %PDF 25 50 44 46 %%EOF
ZIP PK♥♦ 50 4B 03 04 50 4B 05 06 (EOCD)
DOCX ZIP with word/document.xml inside
XLSX ZIP with xl/workbook.xml inside
PPTX ZIP with ppt/presentation.xml inside
⚠️ Note on Office Files

DOCX, XLSX, and PPTX are all ZIP archives internally. They share the same PK\x03\x04 header. To distinguish them, recovery-soft must peek inside the ZIP structure to check for characteristic internal paths.

The FileSignature Struct

Each known file type is described by a FileSignature struct:

type FileSignature struct {
    Name      string   // Human-readable name (e.g., "PDF Document")
    Extension string   // File extension (e.g., ".pdf")
    Header    []byte   // Magic bytes at start of file
    Footer    []byte   // Magic bytes at end of file (nil if none)
    MaxSize   int64    // Maximum expected file size (prevents runaway reads)
}

Key design decisions:

  • Header — Used to detect the start of a file. Must match exactly at a block boundary.
  • Footer — Used to find the end of a file. Not all formats have reliable footers (in which case MaxSize is used as a limit).
  • MaxSize — A safety limit. Without this, a false-positive header match could cause the carver to read gigabytes of garbage data looking for a footer that doesn't exist.

MatchHeader — Byte-by-Byte Comparison

The simplest but most critical function — it checks if a data block starts with a known signature:

func (sig *FileSignature) MatchHeader(data []byte) bool {
    if len(data) < len(sig.Header) {
        return false
    }
    for i, b := range sig.Header {
        if data[i] != b {
            return false
        }
    }
    return true
}

This performs a simple linear comparison. It first checks that the data buffer is at least as long as the signature (prevents out-of-bounds panics), then compares byte by byte. The function returns false at the first mismatch — an early exit optimization that matters when this function is called millions of times during a full disk scan.

FindFooter — Searching Backwards

Once a header is found, the carver needs to find where the file ends. FindFooter searches for the footer pattern within a data region:

func (sig *FileSignature) FindFooter(data []byte) int64 {
    if sig.Footer == nil {
        return -1  // No footer defined for this type
    }

    footerLen := len(sig.Footer)

    // Search backwards from end — footer is most likely near the end
    for i := len(data) - footerLen; i >= 0; i-- {
        match := true
        for j := 0; j < footerLen; j++ {
            if data[i+j] != sig.Footer[j] {
                match = false
                break
            }
        }
        if match {
            return int64(i + footerLen)  // Return position AFTER footer
        }
    }
    return -1  // Footer not found
}

Why search backwards? File footers appear at the end of the file. If we have a buffer containing the entire file, searching from the end finds the footer immediately without scanning through megabytes of content. The function returns the position after the footer, giving us the exact file size.

If the footer isn't found (returns -1), the carver falls back to using MaxSize as the file boundary — this recovers truncated files that may have had their ending overwritten.

The Registry — Known Signatures

All supported file types are registered in a package-level variable:

var Registry = []FileSignature{
    {
        Name:      "PDF Document",
        Extension: ".pdf",
        Header:    []byte{0x25, 0x50, 0x44, 0x46},  // %PDF
        Footer:    []byte{0x25, 0x25, 0x45, 0x4F, 0x46},  // %%EOF
        MaxSize:   100 * 1024 * 1024,  // 100 MB
    },
    {
        Name:      "ZIP Archive",
        Extension: ".zip",
        Header:    []byte{0x50, 0x4B, 0x03, 0x04},  // PK\x03\x04
        Footer:    []byte{0x50, 0x4B, 0x05, 0x06},  // End of central directory
        MaxSize:   500 * 1024 * 1024,  // 500 MB
    },
}

This design makes it trivial to add new file types — just append a new FileSignature to the slice. No other code changes needed. The carver iterates over this registry for every block it reads, checking each signature's header against the block data.

💡 Extending the Tool

To add recovery support for a new file format (e.g., JPEG, PNG), you only need to add its magic bytes to this registry. The scanner, carver, and output packages require zero modifications.

5. The File Carver Engine

The carver (pkg/carver/carver.go) is the heart of raw recovery. It scans every block of the disk looking for known file headers, then extracts data until it finds the corresponding footer.

How Carving Works Step by Step

  1. Open the disk — creates a DiskReader to access raw bytes
  2. Iterate in blocks — reads the disk 4KB at a time (matching typical cluster sizes)
  3. Check for headers — at each position, compares bytes against all known signatures
  4. Extract file data — when a header is found, reads forward up to MaxSize looking for the footer
  5. Classify ZIP files — checks if a ZIP is actually a DOCX/XLSX/PPTX by examining internal paths
  6. Write to output — saves the recovered file with a sequential name

Key Code: The Scan Loop

func (c *Carver) Scan(progressFn func(offset, total int64)) error {
    // Use a sliding window to detect headers that span block boundaries
    overlapSize := 64
    prevTail := make([]byte, 0, overlapSize)

    err := c.reader.ScanBlocks(c.blockSize, func(offset int64, data []byte) bool {
        // Combine previous block's tail with current block's head
        // This catches signatures that span a block boundary
        var searchBuf []byte
        if len(prevTail) > 0 {
            searchBuf = append(prevTail, data...)
        } else {
            searchBuf = data
        }

        // Try every byte position against every known signature
        for i := range searchBuf {
            for sigIdx := range signatures {
                if sig.MatchHeader(searchBuf[i:]) {
                    c.extractFile(sig, actualOffset)
                }
            }
        }
        return true // continue scanning
    })
}

Why the Overlap?

💡 Block Boundary Problem

Imagine a PDF file starts at byte 4094 and your block size is 4096. The %PDF header (4 bytes) would be split: 2 bytes at the end of block 1 and 2 bytes at the start of block 2. Without the overlap buffer, you'd miss it entirely. The 64-byte overlap ensures no header is lost.

File Extraction

func (c *Carver) extractFile(sig *signature.FileSignature, offset int64) {
    // Read data in 1MB chunks, looking for the footer
    for readOffset := int64(0); readOffset < readSize; readOffset += chunkSize {
        chunk := make([]byte, size)
        n, _ := c.reader.ReadAt(chunk, offset+readOffset)
        fileData = append(fileData, chunk[:n]...)

        // Check if footer exists in what we've read so far
        footerEnd := sig.FindFooter(fileData)
        if footerEnd > 0 {
            fileData = fileData[:footerEnd] // Trim to exact file end
            break
        }
    }

    // Classify ZIP-based files (DOCX, XLSX, PPTX)
    if sig.Extension == ".zip" {
        actualSig = c.classifyZip(fileData, sig)
    }

    os.WriteFile(outPath, fileData, 0o644)
}

6. NTFS Filesystem Recovery

The NTFS module (pkg/ntfs/ntfs.go) provides intelligent recovery by parsing the Master File Table — NTFS's equivalent of a file index. This allows recovery with original filenames.

How NTFS Stores Files

NTFS uses a structure called the MFT (Master File Table). Think of it as a database where each row (called an MFT entry) describes one file:

  • Filename — stored in a $FILE_NAME attribute
  • Data location — stored as "data runs" (a list of cluster ranges)
  • Timestamps — creation, modification, access, MFT change
  • Flags — whether the entry is in-use or deleted
⚠️ Why NTFS Recovery Works After Reformatting

When Ubuntu reformats a drive, it writes a new ext4 superblock and inode tables — but these only occupy the first few megabytes. The old NTFS MFT (often located deeper on disk) may survive intact. This is why NTFS mode can recover files with their original names even after a format.

Step 1: Parse Boot Sector

func (p *NTFSParser) ParseBootSector() error {
    buf := make([]byte, 512)
    p.reader.ReadAt(buf, p.partition)

    // Bytes 3-6 must be "NTFS" for valid NTFS
    if string(buf[3:7]) != "NTFS" {
        return fmt.Errorf("not an NTFS filesystem")
    }

    // Extract key geometry values:
    // - Bytes per sector (usually 512)
    // - Sectors per cluster (usually 8, giving 4096-byte clusters)
    // - MFT start cluster number
    p.boot.BytesPerSector = binary.LittleEndian.Uint16(buf[0x0B:0x0D])
    p.boot.SectorsPerCluster = buf[0x0D]
    p.boot.MFTCluster = int64(binary.LittleEndian.Uint64(buf[0x30:0x38]))
}

The boot sector is always at the very start of the partition. It tells us how big clusters are and where the MFT begins.

Step 2: Read MFT Entries

Each MFT entry is exactly 1024 bytes and starts with the magic string FILE:

func (p *NTFSParser) ReadMFTEntry(offset int64) (*MFTEntry, error) {
    buf := make([]byte, 1024) // Every MFT entry is 1KB
    p.reader.ReadAt(buf, offset)

    // Validate signature
    if string(buf[0:4]) != "FILE" { return nil, err }

    // Flags at offset 0x16 tell us:
    // Bit 0: in-use (1) or deleted (0)
    // Bit 1: directory (1) or file (0)
    flags := binary.LittleEndian.Uint16(buf[0x16:0x18])
    entry.InUse = flags&0x01 != 0
    entry.IsDirectory = flags&0x02 != 0

    // Walk through attributes to find filename and data location

}

Step 3: Decode Data Runs

Data runs are a compressed encoding of where file data lives on disk. They use variable-length encoding:

// A data run header byte: low nibble = length size, high nibble = offset size
// Example: 0x31 means 1 byte for length, 3 bytes for offset
//
// Runs are relative: each offset is added to the previous one
// This efficiently encodes fragmented files
func decodeDataRuns(data []byte) []DataRun {
    var prevOffset int64
    for {
        header := data[pos]
        if header == 0 { break } // End of run list

        lengthSize := header & 0x0F  // How many bytes encode the run length
        offsetSize := header >> 4    // How many bytes encode the offset

        // Read length (unsigned) and offset (signed, relative)
        prevOffset += runOffset // Offsets accumulate
        runs = append(runs, DataRun{ClusterOffset: prevOffset, ClusterCount: runLength})
    }
}

Step 4: Recover File Content

func (p *NTFSParser) RecoverFileData(entry *MFTEntry) ([]byte, error) {
    for _, run := range entry.DataRuns {
        // Convert cluster number to byte offset on disk
        clusterOffset := p.partition + run.ClusterOffset * p.boot.ClusterSize
        runSize := run.ClusterCount * p.boot.ClusterSize

        // Read all clusters in this run
        buf := make([]byte, runSize)
        p.reader.ReadAt(buf, clusterOffset)
        data = append(data, buf...)
    }
    // Trim to the file's actual size (last cluster may be partial)
    return data[:entry.FileSize], nil
}

7. ext4 Filesystem Recovery

The ext4 module (pkg/ext4/ext4.go) understands Linux's most common filesystem. It finds deleted files by scanning inode tables for entries with a non-zero deletion timestamp.

How ext4 Organizes Data

ext4 divides the disk into block groups. Each group contains:

  • Superblock — filesystem metadata (block size, inode count, etc.)
  • Group Descriptors — tell you where each group's data structures are
  • Inode Bitmap — marks which inodes are allocated
  • Inode Table — the actual inode entries (like MFT entries in NTFS)
  • Data Blocks — where actual file content lives

Step 1: Parse the Superblock

func (p *Ext4Parser) ParseSuperblock() error {
    // Superblock is always at offset 1024 from partition start
    buf := make([]byte, 1024)
    p.reader.ReadAt(buf, p.partition + 1024)

    // Magic number at offset 0x38 must be 0xEF53
    magic := binary.LittleEndian.Uint16(buf[0x38:0x3A])
    if magic != 0xEF53 {
        return fmt.Errorf("not ext4")
    }

    // Block size is encoded as: 1024 << logBlockSize
    // logBlockSize=0 → 1KB, logBlockSize=1 → 2KB, logBlockSize=2 → 4KB
    logBlockSize := binary.LittleEndian.Uint32(buf[0x18:0x1C])
    p.sb.BlockSize = 1024 << logBlockSize
}

Step 2: Locate Inode Tables via Group Descriptors

func (p *Ext4Parser) ReadGroupDescriptor(groupNum int) (*GroupDescriptor, error) {
    // Group descriptor table starts at block 1 (right after superblock)
    gdtBlock := int64(p.sb.FirstDataBlock) + 1
    gdtOffset := p.partition + gdtBlock * int64(p.sb.BlockSize)

Step 3: Find Deleted Inodes

func (p *Ext4Parser) ScanDeletedInodes(maxGroups int, callback func(*Inode)) error {
    for g := 0; g < numGroups; g++ {
        for i := uint32(0); i < p.sb.InodesPerGroup; i++ {
            inode := p.ReadInode(inodeNum)

            // A deleted file has:
            // - DeletionTime != 0 (was explicitly deleted)
            // - LinkCount == 0 (no directory references it)
            // - Size > 0 (had actual content)
            if inode.IsDeleted() && inode.IsRegularFile() {
                callback(inode)
            }
        }
    }
}

Step 4: Read Data via Extents

Modern ext4 uses extents — contiguous ranges of blocks — instead of indirect block pointers:

8. Usage Guide

Building the Tool

# Clone and build
cd recovery-soft
go build -o recover ./cmd/recover/

# Verify it works
./recover version
# Output: recovery-soft v0.1.0

Recovery Commands

Auto-Detect (Recommended First Try)

# Tries NTFS → ext4 → raw carving automatically
sudo ./recover auto /dev/sda2 /mnt/external/recovered

Raw File Carving

# Works on ANY disk regardless of filesystem
# Best when filesystem is completely destroyed
sudo ./recover carve /dev/sda /mnt/external/recovered

# With verbose output to see each file found
sudo ./recover carve -v /dev/sda /mnt/external/recovered

# Custom block size (use 512 for more thorough but slower scan)
sudo ./recover carve -block-size 512 /dev/sda /mnt/external/recovered

NTFS Recovery (Best for Windows→Linux Scenario)

# Recovers with original filenames!
sudo ./recover ntfs /dev/sda2 /mnt/external/recovered

# If NTFS partition started at an offset
sudo ./recover ntfs -offset 1048576 /dev/sda /mnt/external/recovered

# Scan more MFT entries (for large drives)
sudo ./recover ntfs -max-entries 200000 /dev/sda2 /mnt/external/recovered

ext4 Recovery

# For recently deleted files on an ext4 partition
sudo ./recover ext4 /dev/sda2 /mnt/external/recovered

# Scan more block groups (for large partitions)
sudo ./recover ext4 -max-groups 500 /dev/sda2 /mnt/external/recovered
⚠️ Critical Rules
  • Always use sudo — raw disk access requires root privileges
  • Never write to the source disk — output must go to a different physical drive
  • Act fast — every write to the disk may overwrite recoverable data
  • Boot from USB — if recovering from your main drive, boot a live Linux USB first

10. Troubleshooting

Common Issues

❌ "Permission denied" when accessing disk

Cause: Raw disk access requires root privileges.

Fix:

# Always run with sudo
sudo ./recover carve /dev/sda ./output

# Or check permissions
ls -la /dev/sda
# Should show: brw-rw---- 1 root disk ...
❌ "not an NTFS filesystem"

Cause: The NTFS boot sector was overwritten, or you're pointing at the wrong partition/offset.

Fixes:

  • Try a different partition: /dev/sda1, /dev/sda2, etc.
  • Use fdisk -l to list partitions and their types
  • If the partition table was rewritten, try specifying the old offset manually with -offset
  • Fall back to recover carve — it doesn't need filesystem structures
❌ "not an ext4 filesystem"

Cause: Similar to above — the ext4 superblock isn't where expected.

Fixes:

  • ext4 superblock is at offset 1024 from partition start — verify your offset is correct
  • ext4 stores backup superblocks: try -offset with values from dumpe2fs
  • Fall back to recover carve
❌ Files recovered but corrupted (can't open)

Causes and fixes:

  • Partial overwrite: Some file blocks were reused by the new filesystem. The file is partially destroyed — nothing can recover overwritten bytes.
  • Wrong footer detected: For PDF files, %%EOF can appear mid-file in some cases. The tool uses the last occurrence to minimize this.
  • Fragmented file: In carve mode, fragmented files can't be reassembled (no filesystem metadata). Try ntfs mode instead — it uses data runs to handle fragmentation.
  • SSD TRIM: If the drive is an SSD with TRIM, the hardware may have zeroed the blocks. These zeros produce "valid-looking" headers but empty content.
❌ Recovery finds 0 files

Possible causes:

  • Wrong device: Double-check with lsblk or fdisk -l
  • SSD TRIM: If the drive has been in use since formatting, TRIM may have wiped all deleted data
  • Encrypted drive: BitLocker, LUKS, or FileVault encryption means raw bytes are meaningless without the key
  • Files too small: The tool ignores files smaller than ~20 bytes as false positives
❌ Scan is extremely slow

Fixes:

  • Increase block size: -block-size 65536 (64KB blocks scan faster but may miss some files at unusual alignments)
  • For huge drives, use ntfs or ext4 mode instead of carve — they target specific structures instead of scanning everything
  • USB 2.0 external drives are inherently slow (~30 MB/s). Use USB 3.0+ or direct SATA/NVMe connection.
❌ "failed to determine size" on block device

Cause: The tool uses Seek to find device size. Some unusual devices don't support this.

Fix: Create a disk image first, then recover from the image:

# Create an image of the drive (safe, read-only)
sudo dd if=/dev/sda of=/mnt/external/disk_image.img bs=4M status=progress

# Now recover from the image (can run multiple times safely)
./recover auto /mnt/external/disk_image.img ./recovered

Best Practices for Maximum Recovery

When Recovery Is Impossible

  • Full disk encryption (BitLocker, LUKS, FileVault) without the key
  • Secure erase / zero-fill was performed
  • SSD TRIM has run since deletion (typically within minutes to hours)
  • Drive was heavily used after formatting (all old sectors overwritten)
  • Physical damage — head crash, burned controller, etc. (requires professional data recovery lab)

11. Adding Support for Any File Type

The architecture is designed so adding new file types requires minimal changes. Here's a complete guide to supporting any file format.

Understanding the Extension Pattern

Recovery works by recognizing byte patterns. Every file format has some combination of:

  • Magic header — fixed bytes at the start of the file
  • Footer/trailer — fixed bytes at the end (not all formats have this)
  • Internal structure — size fields, chunk markers, or other length indicators
💡 How to Find Magic Bytes

Use xxd filename | head to see the hex bytes at the start of any file. Compare multiple files of the same type to find the constant prefix. You can also reference the Wikipedia list of file signatures or the file command's magic database at /usr/share/misc/magic.

Advanced: Variable-Offset Headers

Some formats (like MP4) don't start with their magic bytes at offset 0. To handle this, extend the signature system:

type FileSignature struct {
    // ... existing fields ...
    HeaderOffset int  // Where the header appears relative to file start
}

// MP4: "ftyp" appears 4 bytes into the file
{
    Name:         "MP4",
    Extension:    ".mp4",
    Header:       []byte{0x66, 0x74, 0x79, 0x70}, // "ftyp"
    HeaderOffset: 4, // Skip first 4 bytes (size field)
}

// Then modify MatchHeader to account for offset:
func (s *FileSignature) MatchHeader(data []byte) bool {
    start := s.HeaderOffset
    if len(data) < start + len(s.Header) {
        return false
    }
    for i, b := range s.Header {
        if data[start+i] != b { return false }
    }
    return true
}

12. Adding Support for Any Filesystem

To make the tool understand a new filesystem, you implement a parser package that knows how to read that filesystem's metadata structures and locate file data. Here's the complete pattern.

The Filesystem Parser Interface

Every filesystem parser in this tool follows the same logical contract:

// The pattern every filesystem module follows:

// 1. Validate — confirm this disk/partition uses this filesystem
ParseBootSector() / ParseSuperblock()  → error

// 2. Enumerate — find file entries (deleted or active)
ScanEntries(callback)                   → error

// 3. Recover — read the actual file data from disk
RecoverFileData(entry)                  → []byte, error

Step-by-Step: Implementing FAT32 Support

Let's walk through adding FAT32 — used by USB drives, SD cards, and older Windows systems.

1. Create the package

# Create the directory
mkdir pkg/fat32

2. Define the structures

// pkg/fat32/fat32.go
package fat32

import (
    "encoding/binary"
    "fmt"
    "github.com/aneesh/recovery-soft/pkg/scanner"
)

// FAT32 Boot Sector (BPB - BIOS Parameter Block)
type BootSector struct {
    BytesPerSector    uint16
    SectorsPerCluster uint8
    ReservedSectors   uint16
    NumFATs           uint8
    FATSize32         uint32
    RootCluster       uint32 // Usually cluster 2
    ClusterSize       int64
}

// DirectoryEntry represents a 32-byte FAT32 directory entry
type DirectoryEntry struct {
    FileName     string
    Extension    string
    IsDeleted    bool
    IsDirectory  bool
    FileSize     uint32
    StartCluster uint32 // First cluster of file data
}

type FAT32Parser struct {
    reader    *scanner.DiskReader
    partition int64
    boot      *BootSector
    fatStart  int64  // Byte offset of FAT table
    dataStart int64  // Byte offset of data region
}

3. Implement validation

func (p *FAT32Parser) ParseBootSector() error {
    buf := make([]byte, 512)
    p.reader.ReadAt(buf, p.partition)

    // FAT32 identifier at offset 0x52
    fsType := string(buf[0x52:0x5A])
    if fsType != "FAT32   " {
        return fmt.Errorf("not FAT32 (got %q)", fsType)
    }

    p.boot = &BootSector{
        BytesPerSector:    binary.LittleEndian.Uint16(buf[0x0B:0x0D]),
        SectorsPerCluster: buf[0x0D],
        ReservedSectors:   binary.LittleEndian.Uint16(buf[0x0E:0x10]),
        NumFATs:           buf[0x10],
        FATSize32:         binary.LittleEndian.Uint32(buf[0x24:0x28]),
        RootCluster:       binary.LittleEndian.Uint32(buf[0x2C:0x30]),
    }

    // Calculate derived values
    bps := int64(p.boot.BytesPerSector)
    p.boot.ClusterSize = bps * int64(p.boot.SectorsPerCluster)
    p.fatStart = p.partition + int64(p.boot.ReservedSectors)*bps
    fatSize := int64(p.boot.FATSize32) * bps
    p.dataStart = p.fatStart + fatSize*int64(p.boot.NumFATs)

    return nil
}

4. Implement directory scanning

func (p *FAT32Parser) ScanDirectory(cluster uint32, callback func(*DirectoryEntry)) error {
    // Read the cluster containing directory entries
    offset := p.clusterToOffset(cluster)
    buf := make([]byte, p.boot.ClusterSize)
    p.reader.ReadAt(buf, offset)

    // Each directory entry is 32 bytes
    for i := 0; i < len(buf); i += 32 {
        entry := buf[i : i+32]
        if entry[0] == 0x00 { break } // End of directory
        if entry[11] == 0x0F { continue } // Long filename entry (skip)

        de := &DirectoryEntry{
            IsDeleted: entry[0] == 0xE5, // 0xE5 = deleted marker
            FileSize:  binary.LittleEndian.Uint32(entry[28:32]),
        }
        // Start cluster: high 2 bytes at offset 20, low 2 bytes at offset 26
        de.StartCluster = uint32(binary.LittleEndian.Uint16(entry[20:22]))<<16 |
                          uint32(binary.LittleEndian.Uint16(entry[26:28]))

        // Parse 8.3 filename
        de.FileName = strings.TrimRight(string(entry[0:8]), " ")
        de.Extension = strings.TrimRight(string(entry[8:11]), " ")

        callback(de)
    }

    // Follow FAT chain to next cluster of this directory
    nextCluster := p.getNextCluster(cluster)
    if nextCluster < 0x0FFFFFF8 { // Not end-of-chain
        p.ScanDirectory(nextCluster, callback)
    }
    return nil
}

// Convert cluster number to byte offset on disk
func (p *FAT32Parser) clusterToOffset(cluster uint32) int64 {
    // Clusters are numbered starting from 2
    return p.dataStart + int64(cluster-2)*p.boot.ClusterSize
}

// Read FAT table to find next cluster in chain
func (p *FAT32Parser) getNextCluster(cluster uint32) uint32 {
    offset := p.fatStart + int64(cluster)*4
    buf := make([]byte, 4)
    p.reader.ReadAt(buf, offset)
    return binary.LittleEndian.Uint32(buf) & 0x0FFFFFFF
}

5. Implement data recovery

func (p *FAT32Parser) RecoverFileData(entry *DirectoryEntry) ([]byte, error) {
    var data []byte
    cluster := entry.StartCluster

    for cluster > 0 && cluster < 0x0FFFFFF8 {
        offset := p.clusterToOffset(cluster)
        buf := make([]byte, p.boot.ClusterSize)
        p.reader.ReadAt(buf, offset)
        data = append(data, buf...)

        // Follow the FAT chain
        cluster = p.getNextCluster(cluster)
    }

    // Trim to actual file size
    if int64(len(data)) > int64(entry.FileSize) {
        data = data[:entry.FileSize]
    }
    return data, nil
}

6. Register in the CLI

// In cmd/recover/main.go, add a new command:

The Universal Pattern

Regardless of the filesystem, the implementation always follows this flow:

// Every filesystem recovery module:

// 1. IDENTIFY — Read a known fixed location and check for magic bytes
//    NTFS: "NTFS" at offset 3 of sector 0
//    ext4: 0xEF53 at offset 0x438 (1080 bytes from partition start)
//    FAT32: "FAT32   " at offset 0x52
//    HFS+: "H+" at offset 1024

// 2. GEOMETRY — Extract disk layout from the header
//    Block/cluster size, metadata locations, total size

// 3. LOCATE METADATA — Find the file index structure
//    NTFS: MFT (pointed to by boot sector)
//    ext4: Inode tables (pointed to by group descriptors)
//    FAT32: Root directory (at a known cluster)

// 4. ENUMERATE FILES — Walk the index looking for targets
//    Check each entry for: deleted status, file type, size > 0

// 5. FOLLOW DATA MAP — Read file content using the filesystem's mapping
//    NTFS: Data runs (cluster offset + count pairs)
//    ext4: Extents (block offset + length) or block pointers
//    FAT32: Cluster chain (linked list via FAT table)
💡 When Filesystem Recovery Fails

If the filesystem metadata itself was overwritten (new filesystem's structures landing on old metadata), the filesystem-aware parser won't work. That's why carve mode exists as an ultimate fallback — it needs zero metadata and works purely on file content signatures. A robust recovery tool always tries intelligent methods first, then falls back to brute-force carving.

Architecture Tips for New Parsers

  • Keep the package self-contained — only depend on pkg/scanner for disk I/O
  • Handle partial corruption — never assume a structure is intact; always validate before parsing
  • Use DiskReader.ReadAt — it's thread-safe and works on both files and block devices
  • Return errors, don't crash — a single corrupt entry shouldn't abort the entire scan
  • Expose a callback-based scan — this lets the CLI handle output incrementally
  • Test with disk images — use dd to create test images rather than accessing real disks
